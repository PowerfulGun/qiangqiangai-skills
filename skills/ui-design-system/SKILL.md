# UI 设计系统 Skill

当用户需要搭建或遵循一套 UI 设计系统、建立风格指南、或要求界面具备统一的设计语言（如 Linear/Notion/Stripe 式精致风格）时，按以下流程与规范执行。

## 流程概览

1. **定设计方向** — 先确定产品气质与约束，再写代码。
2. **应用工艺原则** — 按下方规则实现布局、间距、色彩、排版。
3. **落风格指南** — 每个组件都在风格指南中有展示与状态说明。

## 执行流程

### 第一步：确定设计方向
- **场景与用户**：产品做什么、谁用、需要传递什么情绪（信任/效率/专注等）。
- **性格取向**（可单选或融合）：
  - 精密与密度 — 紧凑间距、偏单色、信息优先（如 Linear、终端风）。
  - 温暖与易接近 — 宽松间距、柔和阴影（如 Notion、协作工具）。
  - 专业与信任 — 冷色、层次与深度（如 Stripe、企业 B2B）。
  - 鲜明与清晰 — 高对比、留白、果断排版（如 Vercel）。
  - 工具与功能 — 低调配色、清晰层级（如 GitHub、开发者工具）。
- **色彩基础**：暖灰/冷灰/纯中性/带色相，以及是否以深色为主；**主色只用一种**并赋予含义（如蓝=信任、绿=成功）。
- **布局**：内容决定布局 — 信息密集用网格、专注任务用宽松留白、多区块用侧栏/顶栏/分栏列表详情。
- **字体**：系统字体 / 几何无衬线（如 Geist、Inter）/ 人文无衬线 / 等宽元素，与产品调性一致。

### 第二步：工艺原则（必须遵守）
- **4px 网格**：间距以 4 为基数（4/8/12/16/24/32px），统一使用。
- **内边距对称**：同一组件内 TLBR 对称，除非有明确视觉平衡需求。
- **圆角统一**：选定一套（如 4/6/8 或 8/12），不在同一系统内混用多套。
- **深度策略**：在「仅边框」「单层阴影」「多层阴影」「背景色区分」中选一种并统一。
- **控件独立**：日期、筛选、下拉等用定制组件，不直接裸用原生表单控件。
- **排版层级**：标题 600、字间距略紧；正文 400–500；标签 500、可全大写+字间距；字号建议 11/12/13/14(基准)/16/18/24/32。
- **数字与代码**：数字、ID、时间戳用等宽与 `tabular-nums`。
- **图标**：用 Phosphor 或 Lucide，仅在有语义时使用。
- **动效**：微交互约 150ms，较大过渡 200–250ms，缓动可用 `cubic-bezier(0.25, 1, 0.5, 1)`，企业 UI 避免弹跳。
- **色彩语义**：灰色建结构，彩色仅用于状态、操作、成功/错误等语义。
- **深色模式**：多靠边框区分层次，语义色可适当降饱和。

### 第三步：风格指南维护
- 提供或维护一个集中展示页（如 `/style-guide`），包含：色彩、字体、按钮、输入框、卡片、头像、对话框、加载态、设计 token（间距/圆角/阴影）。
- 每个组件展示：默认、悬停、聚焦、加载、成功、错误、禁用、空状态。
- 新组件必须先出现在风格指南中，再在业务中使用；修改组件时同步更新风格指南。
- 风格指南即设计系统的单一事实来源。

### 第四步：避免的做法
- 大范围强阴影、过大的圆角、不对称且无理由的内边距、纯白卡片叠在彩色背景上、过粗装饰性边框、过大块间距（如 >48px）、弹跳动画、装饰性渐变、同一界面多种主色。

## 输出约定

- 若**新建或调整设计系统**：给出设计方向简述、主要 token（间距/圆角/阴影/主色）、以及风格指南的章节结构或对应文件位置。
- 若**实现或修改组件**：说明其符合的网格与深度策略、在风格指南中的位置、以及已展示的状态列表。
- 若**仅做评审**：按上述原则逐条指出不符合项并给出修改建议。
